## Ⅰ. 연구 배경 및 목적 (Research Background)

### 1. 전통적 특허 검토의 한계
전통적인 특허 리스크 스크리닝 및 선행기술 조사는 전문 인력(심사관, 변리사)의 수작업에 의존하여 막대한 시간과 비용이 소모됩니다. 특히, 폭발적으로 증가하는 글로벌 특허 데이터베이스 내에서 출원 발명의 리스크를 선제적으로 파악하는 것은 기업 R&D 전략 수립에 있어 큰 병목 구간입니다.

### 2. 기존 AI 모델의 한계 및 대안
기존의 AI 기반 특허 검색 알고리즘은 단순히 키워드 매칭이나 문장 간의 표면적인 텍스트 유사도 비교에 그쳤습니다. 이로 인해 특허 명세서 특유의 **'용어 회피(Vocab Mismatch)'** 현상을 잡아내지 못하거나, 특허 심사의 핵심인 법리적 판단을 수행하지 못하는 한계가 있었습니다.

### 3. 연구 목적
본 연구는 **프롬프트 파인튜닝(Prompt Fine-tuning)** 기법을 정밀하게 제어하여, LLM이 특허청 심사관의 논리적 사고체계를 모사하도록 유도합니다. 단순 검색을 넘어 특허 법리인 '구성요소 완비의 원칙(All Elements Rule)'과 'TSM(Teaching-Suggestion-Motivation) 모델'을 아키텍처 내부의 로직 게이트로 이식하여, 고차원적인 특허 신규성 및 진보성 판단을 자동화하는 파이프라인을 완비하는 것을 목적으로 합니다.

## Ⅱ. 시스템 아키텍처 및 파이프라인 (System Architecture)

### Step 1: 구성요소 추출 및 구조화 (Pre-processing)
- **독립항 정밀 식별**: 정규식 기반 청구범위 분할 및 LLM 필터링을 통해 인용 관계가 없는 순수 독립항 추출.
- **3요소(Component-Anchor-Role) 구조화**: 하드웨어 명칭에 국한되지 않고, 알고리즘과 데이터 흐름을 기준으로 핵심 동작(Anchor)과 기술적 역할(Role)을 분해하여 정형화.
- **본문 근거(Evidence) 매칭**: `SentenceTransformer`와 `scikit-learn` 코사인 유사도 연산을 통해 추출된 구성요소가 타겟 명세서 상세설명의 어느 문단에 근거하는지 찾아내어 매핑(Grounding)함으로써 LLM의 환각을 차단.

### Step 2: 문맥 기반 정밀 신규성(Novelty) 판단
- **임베딩 기반 시맨틱 검색**: 구조화된 구성요소 정보(Name-Anchor-Role)를 LLM을 통해 최적의 자연어 검색 쿼리로 다각화한 후, `SentenceTransformer`와 `scikit-learn`의 코사인 유사도(`cosine_similarity`) 연산을 통해 수집된 선행문헌 데이터 내에서 의미론적으로 가장 유사한 핵심 문단들을 정밀 추출함.
- **균등론(Doctrine of Equivalents) 기반 대조**: 용어나 표면적 표현이 다르더라도 실질적인 기술적 목적, 기능, 작용 효과가 통상의 기술자 관점에서 쉽게 치환 가능한 균등물(Equivalent)인 경우 `Identical(일치)`로 판정하여 실질적인 신규성 리스크 지수(0~100점)를 산출함.

### Step 3: TSM 모델 기반 진보성(Inventive Step) 추론
- **기술적 컨텍스트 분리**: 발명의 배경, 해결 과제, 수단, 효과 섹션을 헤더 기반으로 명확히 분리 추출.
- **TSM Logic Gate 검증**:
  - **T/S (Teaching/Suggestion)**: 선행문헌 간 결합에 대한 명시적 교시나 암시 검토.
  - **Motivation**: 기술 분야 및 해결 과제의 공통성을 분석하여 복수 선행문헌 간의 결합 동기 판별.
  - **Design Change**: 단순 수치 한정, 재료 선택, 단순 균등물 치환 등 통상의 기술자의 창작 능력 범위 내의 단순 설계 변경인지 판정.
- **효과 현저성 검토**: 선행기술의 단순 결합 예측 범위를 상회하는 질적(이질적 효과)·양적 현저성이 있는지 최종 대조.

> 단순히 단어나 문맥의 표면적 유사도를 비교하는 기존 AI 검색 시스템을 넘어, **'구성요소 완비의 원칙'**, **'균등론'**, **'TSM 모델'** 등 실제 특허 심사관의 복잡한 법리적 사고 체계를 AI 아키텍처 내에 투영해 냈습니다.

### 💡 핵심 연구 성과 (Key Accomplishments)
* **특허 특화 구조화 파이프라인 정립 (`Step 1`)**
  * 정규식 기반 분할과 LLM 필터링의 하이브리드 방식을 적용, 복잡한 청구범위 내에서 순수 독립항을 **100% 정밀 식별**함.
  * AI/소프트웨어 발명에 최적화된 **`[구성요소(Component) - 핵심동작(Anchor) - 기술적 역할(Role)]`** 구조를 확립하여 정형 데이터화 성공.
  * 규칙 기반 가중치 스코어링 알고리즘을 통해 명세서 본문 내 정확한 단락 번호(`【0000】`)를 추적하고 근거 문장을 매핑(**Grounding**)하여 LLM 환각을 원천 차단함.
* **균등론 기반의 정밀 신규성 판단 (`Step 2`)**
  * 외부 벡터 DB 프레임워크에 의존하지 않고, `SentenceTransformer` 밀집 벡터 공간에서 **코사인 유사도를 직접 연산**하여 통제력 높은 의미론적 검색을 수행함.
  * LLM 유의어 확장 레이어를 구축하여 표면적 단어는 다르지만 실질적 목적·효과가 동일한 균등물(Equivalent)을 추적, `Identical` / `Modified` / `Distinct`로 세밀하게 등급을 분류함.
* **TSM 법리 모델 기반의 진보성 추론 자동화 (`Step 3`)**
  * 대한민국 특허 심사 기준인 **TSM(Teaching-Suggestion-Motivation)** 모델을 LLM 내부 로직 게이트로 성공적으로 구현함.
  * 문헌 간 해결 과제의 공통성을 분석해 결합 동기를 적출하고, 단순 결합 대비 **효과의 현저성(Predictability)** 여부를 엄격하게 판정함.

## Ⅳ. 연구 한계점 및 향후 과제 (Limitations & Future Work)

* 🖼️ **멀티모달 데이터 처리의 한계**: 특허 명세서에서 비중이 매우 큰 '도면 이미지 데이터(Vision)'를 구조적으로 파싱하고 본문 텍스트와 결합하여 분석하는 멀티모달 파이프라인의 부재.

* 🎯 **향후 계획 (Next Roadmap)**: 
  * 차기 버전에서는 멀티모달 LLM 아키텍처(GPT-4o Vision 등)를 도입하여 도면 분석 모듈을 파이프라인에 결합할 예정.
  * 일반 한국어 모델을 넘어, 특허 도메인 코퍼스를 활용한 **특허 전용 임베딩 모델 미세조정(Domain-Specific Fine-tuning)**을 수행하여 시맨틱 검색 성능을 극대화할 계획.

### 2. 환경 변수 및 초기 세팅 (Environment Setup)
* 본 프로젝트는 분석의 편의를 위해 Google Colab 환경에 최적화되어 있습니다. 소스코드 상단의 초기화 셀을 통해 Google Drive를 마운트하고, OpenAI API 키를 안전하게 입력한 뒤 데이터셋 경로를 지정해 줍니다.

```python
import os
import getpass
from google.colab import drive
from langchain_openai import ChatOpenAI

# 1. Google Drive 마운트
if not os.path.exists('/content/drive'):
    drive.mount('/content/drive')

# 2. OpenAI API 키 입력 (getpass를 통해 화면 노출 방지)
if "OPENAI_API_KEY" not in os.environ:
    print("\n🔑 OpenAI API Key를 입력하세요:")
    os.environ["OPENAI_API_KEY"] = getpass.getpass()

# 3. 데이터셋 루트 경로 설정 (본인의 드라이브 환경에 맞게 수정)
DATASET_ROOT = "YOUR_GOOGLE_DRIVE_DATASET_PATH"

# 4. LLM 분석 엔진 초기화 (GPT-4o)
try:
    llm = ChatOpenAI(model_name="gpt-4o", temperature=0, seed=42)
    print("✅ LLM (gpt-4o) 분석 엔진 초기화 성공")
except Exception as e:
    print(f"❌ LLM 초기화 실패: {e}")
```
